# 根因

## QA-501 [HIGH, post_fix_defect] S-10 invariant 違反
runner.py:1508 で Gate 反転時に gate_check["passed"] (後方互換 alias) と
gate_check["overall_passed"] (canonical) の両方を同期するが、9c83b9f5 で導入した
backfill スクリプトはこの同期を欠いていた。結果として 15 件中 11 件で
`gate_check.passed=True / overall_passed=False / failed_conditions=[]` の不整合が
残存し、oncall が gate_check を直読みすると「Gate PASS なのに overall FAIL ?」と
混乱する状態だった。

第1チーム実測 (QA-501): 3 戦略 (0506-026, 0508-037, 0509-005) の result.json で
S-10 invariant 違反を確認。FTS Live 影響は result_parser.py:L211-216 の AND
合成で 0 だが、将来この保護が削除された場合に顕在化するリスク。

## QA-503 [HIGH, post_fix_defect] H-NEW-1 (layer2.passed) invariant 違反
runner.py:1443/1477/1502 は overall_passed 反転時に layer2.passed も同期するが
9c83b9f5 の backfill はこの同期を欠いていた。event_simulator 側の緩い passed=True
が信頼されるべきでない (H-E 第4次監査) のに、backfill 後の result.json で
`layer2.passed=True / overall_passed=False` の anti-pattern が再現していた。

# 修正内容

## scripts/backfill_v6_2_0_soft_score.py 更新

- **QA-501 修正**: `gate_check["passed"] = new_overall_passed` を同期 (新規)
- **QA-503 修正**: `layer2["passed"] = new_overall_passed` を同期 (新規、error=None 時のみ)
- **failed_conditions marker** を追加 (oncall 判断材料):
  - Tier2 FAIL: `backfill_v6_2_0_soft_score_below_floor`
  - is_duplicate 反転: `cluster_cap_exceeded` (runner.py:1513 と同じマーカー)
  - その他: `backfill_v6_2_0_tier1_or_other_failure`
- **--force フラグ**: METRICS_SCHEMA_VERSION=6.2.0 の戦略も再処理可能化
- **--gids-from <FILE> フラグ**: 改行区切り GID ファイルから対象を読込
